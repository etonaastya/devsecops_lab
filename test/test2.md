**1. Атакующий, получивший доступ к процессу с правами root, читает файл /proc/<pid>/maps целевого процесса. Зачем?**
* **Правильный ответ:** C) Для определения адресов и смещений областей памяти процесса — необходимо перед чтением /proc/<pid>/mem для дампа секретов из памяти (ключи, пароли, токены)
* **Обоснование:** Файл `/proc/<pid>/maps` содержит карту виртуальной памяти процесса, где указаны начальные и конечные адреса, права доступа и привязка к файлам. Без этой карты чтение `/proc/<pid>/mem` бессмысленно, так как это разреженный файл с "дырами" (неотображёнными областями). Зная точные смещения, атакующий может извлечь из памяти конкретные структуры (ключи шифрования, сессии, токены), не парся гигабайты нулевых байт.

**2. Что включает флаг -A при запуске Nmap?**
* **Правильный ответ:** D) Определение ОС, версий сервисов, traceroute и запуск NSE-скриптов
* **Обоснование:** Флаг `-A` (Aggressive) является макросом, который одновременно активирует `-O` (определение ОС), `-sV` (определение версий сервисов), `-sC` (запуск стандартных NSE-скриптов) и `--traceroute`. Он не меняет количество сканируемых портов (для этого используется `-p-`) и не управляет таймаутами напрямую.

**3. Для чего используется multi-stage build в Dockerfile?**
* **Правильный ответ:** B) Для уменьшения итогового размера образа путём исключения build-зависимостей
* **Обоснование:** Multi-stage build позволяет использовать несколько инструкций `FROM`. На промежуточных этапах можно применять тяжёлые образы с компиляторами, SDK и пакетными менеджерами для сборки проекта. Финальный образ начинается с минимального рантайма, куда копируются только скомпилированные бинарные файлы и статические ресурсы. Это исключает из продакшн-образа все промежуточные артефакты, кеши пакетов и инструменты разработки.

**4. Какой риск несёт флаг --privileged при запуске Docker-контейнера?**
* **Правильный ответ:** C) Контейнер получает полный доступ к ресурсам хоста, упрощая побег из изоляции
* **Обоснование:** Флаг `--privileged` отключает AppArmor/SELinux профили, сбрасывает cgroups-ограничения и предоставляет контейнеру все Linux capabilities, а также прямой доступ ко всем блочным и символьным устройствам хоста (`/dev`). Это фактически превращает контейнер в процесс с правами root на хосте, делая изоляцию файловой системы и ядра номинальной и значительно упрощая container escape.

**5. Чем Semgrep принципиально отличается от обычного grep?**
* **Правильный ответ:** B) Semgrep анализирует языковые конструкции (AST), а не просто строки текста
* **Обоснование:** `grep` оперирует регулярными выражениями над плоским текстом, не понимая синтаксис языка. Semгреп парсит исходный код в абстрактное синтаксическое дерево (AST) и сопоставляет запросы с семантической структурой кода. Это позволяет выявлять уязвимости, зависящие от контекста (например, передача непроверенных данных в SQL-запрос или небезопасная десериализация), которые невозможно надежно найти текстовым поиском.

**6. Какую уязвимость эксплуатирует запрос /search?username=admin' OR '1'='1?**
* **Правильный ответ:** C) SQL Injection
* **Обоснование:** Конструкция `' OR '1'='1` является классическим пейлоадом для внедрения в параметр запроса, который модифицирует логику SQL-запроса на сервере. Вставка закрывает строковый литерал и добавляет условие, которое всегда истинно. Это позволяет обойти проверку аутентификации или извлечь данные, что является прямым индикатором уязвимости SQL Injection.

**7. Чем CWE отличается от CVE?**
* **Правильный ответ:** C) CWE — классификатор типов слабостей кода; CVE — идентификатор конкретной уязвимости
* **Обоснование:** CWE (Common Weakness Enumeration) — это каталог типов программных недостатков и архитектурных слабостей (например, «Недостаточная проверка входных данных», «Использование устаревших библиотек»). CVE (Common Vulnerabilities and Exposures) — это реестр конкретных, публично раскрытых уязвимостей в реальных продуктах, каждая из которых имеет уникальный номер (например, CVE-2024-XXXX). Одна категория CWE может описывать сотни различных CVE в разных продуктах.

**8. Разработчик работает над feature-веткой... После hotfix он делает git stash pop и получает merge-конфликт. Какой вариант действий корректен?**
* **Правильный ответ:** B) Разрешить конфликт вручную, затем git stash drop — stash не удаляется автоматически при конфликте в pop, без drop он останется в стеке
* **Обоснование:** Поведение `git stash pop` при конфликте специфично: изменения применяются в рабочую директорию, но запись из стека **не удаляется**, чтобы избежать потери данных. Разработчик должен вручную разрешить конфликты, выполнить `git add` и коммит, а затем явно удалить запись из стека командой `git stash drop` (или `git stash pop` уже не отработает без флага `--index`). Варианты с `drop` до разрешения или `--force` приведут к потере изменений или некорректному состоянию репозитория.

**9. Checkov сканирует docker-compose.yml и выдаёт предупреждение... Разработчик отключает проверку через # checkov:skip=CKV_DOCKER_2. Что принципиально не так с таким подходом?**
* **Правильный ответ:** C) Замалчивание предупреждения не устраняет уязвимость: контейнер всё равно запущен с --privileged, CI будет зелёным, но риск побега из изоляции остаётся — это ложное ощущение безопасности
* **Обоснование:** Коммент `# checkov:skip=` влияет исключительно на вывод статического анализатора конфигураций. Он не изменяет поведение Docker или runtime-среды. Контейнер продолжит запускаться в режиме `--privileged` с полным набором прав. Пропуск проверки без бизнес-обоснования и компенсирующих мер создаёт ложное чувство безопасности, так как реальная поверхность атаки не уменьшается, а только скрывается в отчётах.

**10. Какие из следующих настроек в sshd_config наиболее критичны для hardening SSH-сервера на production-хосте?**
* **Правильный ответ:** B) Установить PermitRootLogin no, PasswordAuthentication no, AllowUsers, ограничить MaxAuthTries
* **Обоснование:** Указанные параметры соответствуют рекомендациям CIS Benchmarks и NIST. `PermitRootLogin no` исключает прямой брутфорс учётной записи суперпользователя. `PasswordAuthentication no` принуждает использовать криптографические ключи, устойчивые к подбору. `AllowUsers` реализует принцип минимальных привилегий на уровне доступа. `MaxAuthTries` ограничивает количество попыток входа, замедляя автоматизированные атаки. Остальные варианты либо снижают безопасность (D), либо являются недостаточными (C), либо ухудшают отказоустойчивость (A).
