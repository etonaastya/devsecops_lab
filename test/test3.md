**1. Команда git push --force отклонена — на удалённой ветке main включена защита (branch protection). Разработчик хочет переписать историю (удалить коммит с секретом). Какой подход корректен?**
* **Правильный ответ:** C) Создать новую ветку с очищенной историей через git filter-repo, открыть PR — branch protection не обходится, история переписывается через штатный процесс ревью
* **Обоснование:** Правила защиты веток (branch protection) запрещают прямые force-push и перезапись истории в критичных ветках. Обход этих правил отключением защиты нарушает политики безопасности и аудита. Корректный workflow предполагает очистку истории в отдельной ветке с помощью `git filter-repo` (или BFG Repo-Cleaner), пуш этой ветки и создание Pull Request. Слияние через PR сохраняет требования к код-ревью, поддерживает цепочку аудита и гарантирует, что защищённая ветка обновится только после проверки.

**2. Какой сигнал отправляет команда kill -9 <PID>?**
* **Правильный ответ:** C) SIGKILL — принудительное немедленное завершение
* **Обоснование:** Сигнал `-9` соответствует `SIGKILL`. В отличие от `SIGTERM` (15), который процесс может перехватить и обработать, `SIGKILL` обрабатывается напрямую ядром ОС. Он немедленно прекращает выполнение процесса без вызова обработчиков завершения, закрытия файловых дескрипторов или освобождения ресурсов, поэтому используется только как крайняя мера при зависании процессов.

**3. Nmap сканирует хост: порт 8080/tcp open http-proxy. Какой следующий шаг наиболее информативен для оценки поверхности атаки через этот сервис?**
* **Правильный ответ:** B) Запустить nmap -sV --script=http-methods,http-headers,http-title -p 8080 <host> — определить версию ПО, разрешённые методы и заголовки безопасности
* **Обоснование:** После обнаружения открытого TCP-порта с HTTP-сервисом стандартная методология требует верификации версии (`-sV`) и сбора прикладных метаданных. Скрипты `http-methods` выявляют опасные HTTP-методы (PUT, DELETE, TRACE), `http-headers` проверяют наличие защитных заголовков (CSP, HSTS, X-Frame-Options), а `http-title` даёт контекст назначения сервиса. Это формирует базу для дальнейшей оценки уязвимостей. Сканирование UDP на TCP-порту бессмысленно, а повторные SYN-сканы не добавляют прикладной информации.

**4. Контейнер в production потребляет 100% CPU и вызывает деградацию соседних сервисов на том же хосте. В docker-compose.yml нет ограничений ресурсов. Какая конфигурация корректно решит проблему?**
* **Правильный ответ:** B) cpus: "0.5" и mem_limit: 512m через deploy.resources.limits — cgroups v2 ограничит контейнер 50% одного ядра и 512 МБ RAM
* **Обоснование:** По умолчанию Docker не устанавливает жёстких лимитов на CPU и память, что позволяет одному контейнеру монополизировать ресурсы хоста. Директивы `cpus` и `mem_limit` (или `deploy.resources.limits` в современных версиях Compose) транслируются в правила Linux cgroups, которые гарантируют изоляцию ресурсов. Ограничение памяти не влияет на использование CPU, а настройка рестарта не решает проблему конкуренции за вычислительные мощности.

**5. Trivy сканирует образ и выдаёт 47 CVE, из которых 3 критических. Все критические — в пакете libssl3 базового образа debian:bookworm-slim. Команда не может обновить базовый образ из-за совместимости. Какой подход снижает риск без смены базового образа?**
* **Правильный ответ:** B) RUN apt-get update && apt-get upgrade -y libssl3 в Dockerfile — пропатчить только уязвимый пакет внутри образа
* **Обоснование:** Когда обновление базового образа невозможно из-за требований совместимости зависимостей, стандартом является целевое обновление конкретных уязвимых пакетов на этапе сборки. Команда `apt-get upgrade` для `libssl3` загрузит последние патчи безопасности из репозитория Debian, закрыв известные CVE без изменения версии ОС. Игнорирование уязвимостей через `.trivyignore` или изменение фильтров сканера не снижает реальную поверхность атаки.

**6. На какой тип файлов в первую очередь направлен инструмент Checkov?**
* **Правильный ответ:** B) Infrastructure as Code (Terraform, Dockerfile, Kubernetes YAML)
* **Обоснование:** Checkov — это статический анализатор конфигураций, разработанный специально для Infrastructure as Code (IaC). Он проверяет манифесты и скрипты развёртывания (Terraform, CloudFormation, Kubernetes, Dockerfile, Helm, Ansible, CloudFormation) на соответствие лучшим практикам безопасности, нормативным требованиям (CIS, NIST, GDPR) и выявляет опасные настройки до применения инфраструктуры. Он не предназначен для анализа исходного кода прикладных программ.

**7. Что обеспечивает атрибут HttpOnly у cookie?**
* **Правильный ответ:** B) Запрещает доступ к cookie через JavaScript, защищая от XSS-кражи
* **Обоснование:** Флаг `HttpOnly` указывает браузеру ограничивать передачу файла cookie исключительно через HTTP/HTTPS запросы, делая его недоступным для чтения и модификации через клиентские скрипты (`document.cookie`). Это является ключевым механизмом защиты сессионных токенов от хищения при успешной эксплуатации Cross-Site Scripting (XSS). Флаг не влияет на шифрование передачи (за это отвечает `Secure`) и не управляет временем жизни.

**8. Что такое Threat Modeling (моделирование угроз)?**
* **Правильный ответ:** B) Анализ возможных угроз и векторов атак на этапе проектирования системы
* **Обоснование:** Моделирование угроз — это проактивная, структурная практика, проводимая на стадии архитектуры и проектирования. Она включает идентификацию ценных активов, определение границ доверия, систематический перебор потенциальных векторов атак (часто с использованием методологий STRIDE или PASTA) и проектирование компенсирующих контролей до начала разработки. Это отличает его от реактивного тестирования (DAST, пентесты) или автоматизированного сканирования кода.

**9. В docker-compose.yml смонтирован том /var/run/docker.sock:/var/run/docker.sock внутрь контейнера приложения. Какую критическую угрозу это создаёт?**
* **Правильный ответ:** B) Контейнер получает возможность создавать новые контейнеры с флагом --privileged и монтированием файловой системы хоста — фактически полный побег из изоляции
* **Обоснование:** `/var/run/docker.sock` — это UNIX-сокет, через который приложения взаимодействуют с Docker Daemon API. Его монтирование внутрь контейнера предоставляет этому контейнеру права суперпользователя Docker. Компрометация такого контейнера позволяет атакующему через Docker API запускать новые контейнеры с `--privileged`, монтировать корневую ФС хоста (`-v /:/host_root`) и выполнять произвольный код на уровне ОС хоста, полностью нивелируя изоляцию контейнеров.

**10. Компания хранит данные клиентов в self-hosted PostgreSQL. После аудита выявлен риск утечки из-за отсутствия шифрования at-rest и слабого backup. Бюджет ограничен. CTO предлагает мигрировать в managed AWS RDS с encryption и автобэкапами. Какая стратегия управления риском применяется?**
* **Правильный ответ:** C) Risk Transfer — ответственность за инфраструктурную безопасность (шифрование, бэкапы, патчинг ОС) передаётся AWS, но ответственность за данные и конфигурацию остаётся на компании
* **Обоснование:** Переход на управляемое облачное решение в рамках модели разделённой ответственности (Shared Responsibility Model) является классическим примером переноса риска (Risk Transfer). Провайдер берёт на себя обязательства по защите физической инфраструктуры, гипервизора, шифрованию хранилищ, резервному копированию и обновлению ОС. Организация сохраняет ответственность за управление доступом, классификацию данных, настройку параметров БД и безопасность прикладного уровня. Это снижает операционные риски и перекладывает часть финансовой и технической ответственности на вендора в рамках SLA.
