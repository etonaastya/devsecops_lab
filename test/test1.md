1. Какому файлу конфигурации Git соответствует уровень --global?
Правильный ответ: C) ~/.gitconfig
Обоснование: В иерархии конфигурации Git уровень --global отвечает за настройки текущего пользователя и сохраняется в файле ~/.gitconfig (или ~/.config/git/config в новых версиях). Уровень --system соответствует /etc/gitconfig, а --local (применяется по умолчанию) — .git/config внутри конкретного репозитория.
2. Флаг setuid (chmod u+s) установлен на исполняемый файл, принадлежащий root. В чём состоит угроза безопасности?
Правильный ответ: B) Файл исполняется с правами владельца (root) независимо от того, кто его запустил — уязвимость в нём позволяет обычному пользователю получить root-привилегии
Обоснование: Бит setuid указывает ядру ОС запускать процесс с UID владельца файла, а не вызывающего пользователя. Если владелец root, любой локальный пользователь получает выполнение кода с максимальными привилегиями. Наличие ошибки (например, переполнение буфера, race condition или некорректная обработка путей) в таком бинарнике напрямую ведёт к повышению привилегий (Privilege Escalation).
3. Как называется тип сканирования Nmap, отправляющий SYN-пакеты без завершения TCP-рукопожатия?
Правильный ответ: D) TCP SYN/Stealth scan (-sS)
Обоснование: При сканировании SYN (-sS) Nmap отправляет пакет SYN и ожидает ответ: SYN-ACK указывает на открытый порт, RST — на закрытый. После получения ответа Nmap отправляет RST, разрывая соединение до завершения трёхэтапного рукопожатия. Это ускоряет сканирование и снижает вероятность фиксации сессии в системных логах сервера по сравнению с полным connect-сканированием (-sT).
4. Разработчик собирает Docker-образ... Итоговый образ весит 1.2 ГБ. Какой подход уменьшит размер до ~150 МБ без потери функциональности?
Правильный ответ: C) Использовать multi-stage build: собрать зависимости в python:3.11, скопировать артефакты в python:3.11-slim или alpine
Обоснование: Multi-stage сборка позволяет использовать тяжёлый образ с инструментами компиляции и зависимостями только на этапе build, а в финальный образ копировать исключительно готовые артефакты и минимальный рантайм. Замена базового образа на slim или alpine кардинально снижает размер. Очистка кэша в том же слое RUN не уменьшает итоговый размер образа из-за особенностей слоёвой файловой системы Docker.
5. В CI/CD пайплайне Semgrep (SAST) не нашёл уязвимостей, но OWASP ZAP (DAST) обнаружил SQL Injection. Какое объяснение наиболее вероятно?
Правильный ответ: C) Уязвимость возникает в runtime из-за динамической конкатенации запросов через ORM или хранимые процедуры, которые SAST не видит без контекста выполнения
Обоснование: SAST-инструменты анализируют статический код и часто не могут корректно отследить сложные цепочки передачи данных, динамическую генерацию запросов в runtime или особенности работы фреймворков/библиотек. DAST проверяет работающее приложение, отправляя реальные тестовые пейлоады, поэтому способен выявить уязвимости, проявляющиеся только при фактическом выполнении кода и взаимодействии с БД.
6. Как называется уязвимость, при которой вредоносный скрипт выполняется в браузере жертвы через отражённые входные данные?
Правильный ответ: D) Reflected XSS
Обоснование: Reflected (отражённая) Cross-Site Scripting возникает, когда вредоносный скрипт передаётся в запросе (например, через URL-параметр или поле формы), немедленно отражается сервером в HTML-ответе и выполняется в контексте браузера жертвы. В отличие от Stored XSS, код не сохраняется на сервере, а существует только в рамках одного запроса-ответа.
7. В GitHub Actions workflow секрет передаётся в шаг через env... Разработчик добавил шаг run: echo $TOKEN для отладки. Какой риск это создаёт?
Правильный ответ: D) При некоторых условиях (set -x, перенаправление вывода) секрет может попасть в артефакты или сторонние action, логи которых не маскируются
Обоснование: Механизм маскирования секретов в GitHub Actions работает путём поиска точных совпадений строк в стандартном выводе шагов. Он не является абсолютно надёжным: при включённом режиме отладки (set -x), кодировании вывода, использовании сторонних действий или сохранении логов в артефакты секрет может отобразиться в открытом виде. Явный вывод секрета в консоль нарушает принципы безопасной разработки и повышает риск утечки.
8. Trivy сканирует production-образ и находит CVE-2024-21626 (runc container escape, CVSS 8.6)... securityContext.runAsNonRoot: true. Какой приоритет действий?
Правильный ответ: B) Немедленно обновить базовый образ до пропатченной версии — container escape обходит userspace-ограничения, runAsNonRoot не защищает от уязвимости в runc
Обоснование: Уязвимости уровня container runtime (такие как в runc) позволяют атакующему выйти за пределы изоляции контейнера на уровне хостовой ОС. Настройка runAsNonRoot применяется внутри контейнера и ограничивает запуск процессов от UID 0, но не влияет на механизмы запуска и изоляции на уровне ядра и рантайма. Поскольку уязвимость критическая и позволяет обойти изоляцию, требуется срочное обновление.
9. Что означает стратегия управления риском Risk Acceptance?
Правильный ответ: C) Организация осознанно решает не принимать защитные меры
Обоснование: Risk Acceptance (принятие риска) — это формализованное решение руководства или владельца актива не внедрять дополнительные контрмеры. Оно принимается после анализа вероятности и потенциального ущерба, когда стоимость защиты превышает ожидаемые потери, либо когда риск находится в пределах установленной толерантности организации. Риск не устраняется и не передаётся, а осознанно остаётся на балансе организации.
10. Разработчик случайно закоммитил файл .env с секретами и сразу сделал git push. Какова первоочерёдная правильная последовательность действий?
Правильный ответ: C) Немедленно ротировать скомпрометированные секреты, затем очистить историю через git filter-repo или BFG, затем force-push — секрет уже мог быть считан сразу после push
Обоснование: Как только секрет попадает в удалённый репозиторий, он считается скомпрометированным, поскольку любой пользователь с доступом к репозиторию, а также системы кэширования, зеркала или CI/CD-агенты могли его зафиксировать. Первым и обязательным шагом является ротация (отзыв и генерация новых) учётных данных. Только после нейтрализации угрозы следует очистить историю коммитов специализированными утилитами и принудительно перезаписать ветку. Удаление файла новым коммитом или git revert не удаляет секрет из истории и не защищает от его использования.
